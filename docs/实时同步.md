# 实时同步机制分析报告

## 1. 现状分析：为何无法实时显示？

通过查阅代码 (`web/src/components/canvas/MapCanvas.tsx`, `web/src/hooks/useSync.ts`, `cf-workers/src/DrawingRoom.ts`)，目前的同步机制如下：

1.  **基于 "瓦片" 的房间机制**：
    *   前端 `useSync` 会根据当前地图中心点，计算出一个 **Zoom 14 级别的瓦片坐标** (例如 `14/13358/6182`) 作为 "房间号"。
    *   WebSocket 连接会加入这个特定的房间。
    *   **问题核心**：如果您和另一台设备的地图中心点不在同一个 Zoom 14 瓦片内（即使你们屏幕上看到的是同一个区域），你们会被分配到 **不同的 WebSocket 房间**。
    *   **结果**：设备 A 的画笔事件只会广播给同房间的设备。设备 B 如果中心点稍有偏移（在相邻瓦片），就收不到广播，必须移动地图触发 `loadViewport` 走 HTTP API 拉取数据才能看到。

2.  **数据流向**：
    *   **实时** (WebSocket): 仅在同一房间（同一瓦片中心）内的用户之间广播。
    *   **持久化** (HTTP API): 画笔数据会同时写入数据库。当用户移动地图时，触发 `handleViewportChange` -> `loadViewport`，通过 API 拉取当前视野内的所有数据。这就是为什么 "刷新" 或 "移动地图" 能看到的根本原因。

**结论**：目前支持实时同步，但 **触发条件极其苛刻**（必须两人中心点非常接近，处于同一瓦片）。对于地图应用来说，这种 "单点房间" 设计是不符合直觉的，导致用户体验为 "不实时"。

## 2. 产品设计分析：是否有必要实时显示？

### 方案对比

| 特性 | 方案 A：严格实时同步 (WebSocket/DO) | 方案 B：移动/刷新时同步 (Polling/Fetch) |
| :--- | :--- | :--- |
| **用户体验** | **极佳**。对方画一笔，我立刻看到。适合“协同作业”、“会议演示”、“即时指挥”等场景。 | **一般/延迟**。对方画完，我需要动一下地图才出现。适合“异步留言”、“静态标记”场景。 |
| **技术复杂度** | **高**。需要解决跨瓦片广播问题 (QuadTree / PubSub) 或订阅多个相邻瓦片房间。 | **低**。目前的 API 拉取模式已经跑通，且逻辑简单。 |
| **服务器负载** | 持续的长连接，高频小包传输。 | 突发性 API 请求。 |
| **适用场景** | 强交互、多人同时在线协作。 | 弱交互、单人记录、事后查看。 |

### 建议
*   **如果不符合早期设计需求**，说明产品定位偏向 **“实时协作”**（如在线白板、协同指挥）。此时 **必须** 优化实时性。目前的 "移动刷新" 体验会打断用户心流，让用户产生 "还没同步过来吗？" 的疑惑，属于 **Bug 级体验**。
*   **如果** 产品定位仅是 "地图留言板"，那么移动刷新是可以接受的，但在 UI 上需要给用户明确提示（如 "有新内容，点击刷新"）。

**综合判定**：既然您提到 "不符合早期设计需求"，建议 **必须修复实时性**。

## 3. 资源耗用与 Cloudflare 免费额度分析

主要涉及 CloudflareWorkers (API) 和 Durable Objects (WebSocket 房间)。

### 方案 A：优化后的实时同步 (推荐)
*   **机制**：用户订阅当前视野覆盖的所有瓦片（例如 3x3 = 9 个房间），或者后端做广播路由。
*   **Cloudflare 成本**：
    *   **Durable Objects (DO)**: 只要使用了 DO，通常需要 **Workers Paid Plan ($5/月)**。免费版对 DO 的支持有限（主要是测试用）。
    *   **请求数**：WebSocket 建立连接算 1 次请求。后续消息传输不算请求数（但在 DO 内部计算 CPU 时间）。只要连接保持，消耗极低。
    *   **免费额度**：如果必须用免费版，DO 是最大瓶颈。目前 Cloudflare 免费版已开放一定的 DO 额度用于测试，但生产环境建议上 Paid Plan。

### 方案 B：移动/轮询刷新 (现状)
*   **机制**：每次移动地图调用 API。如果为了模拟实时，改为 "每 5 秒自动轮询"。
*   **Cloudflare 成本**：
    *   **Workers**：免费版每天 **10 万次请求**。
    *   **计算**：
        *   若 100 个在线用户，每人每 10 秒轮询一次。
        *   每分钟 6 次 x 60 分钟 x 24 小时 = 8640 次/人/天。
        *   100 人 = 86.4 万次/天 -> **远超免费额度**。
    *   **结论**：用轮询模拟实时，**反而更贵** 且容易撞墙。移动刷新（被动触发）虽然省流，但体验差。

### 结论
*   **Cloudflare 免费版**：很难完美支撑高频实时协作（因为 DO 的限制或轮询的请求量限制）。
*   **性价比首选**：升级到 Cloudflare Workers Paid ($5/月)，包含 1000 万请求和更好的 DO 支持。这对于商业或严肃项目来说成本几乎可以忽略不计。
*   **如果不升级**：目前的 "移动刷新" 是唯一能在免费额度下长期稳定运行的方案，但牺牲了体验。

## 修复建议 (若决定优化实时性)

不需要重构整个架构，只需修改 **前端订阅逻辑**：

1.  **多房间订阅**：
    *   计算当前视野覆盖了哪些 Zoom 14 瓦片（通常屏幕可能覆盖 1-4 个瓦片）。
    *   `useSync` 同时加入这些房间。
    *   这样无论对方在哪个瓦片画，只要在你的视野内，你都能收到广播。

2.  **后端轻微调整**：
    *   无需大改。`DrawingRoom` 维持现状即可。


此方案能以最小代价实现真正的实时同步。

## 4. 进阶方案：既省资源又体验好的 "瓦片化缓存策略"

针对您提到的 **"移动地图请求太频繁，浪费资源"** 这一痛点，我们推荐采用 **"瓦片化按需加载 + 客户端缓存"** 方案。这也是 Google Maps、Mapbox 等成熟地图产品的标准做法。

### 核心逻辑

1.  **化整为零 (Tile-based Fetching)**
    *   **现状**：目前的 API 是传 `minLat, maxLat` (视野边界)。用户稍微挪动一点地图，边界就变了，必须重新请求。
    *   **优化**：将地图切分为固定的网格（瓦片，Tile）。例如 Zoom 14 下，屏幕可能包含 `Tile(x=1,y=1)` 到 `Tile(x=3,y=3)` 这 9 块。
    *   **请求方式**：前端不再传边界，而是计算出当前视野需要哪些 **瓦片坐标** (如 `x=1&y=1&z=14`)。

2.  **只请求 "未知" 区域**
    *   前端维护一个 `LoadedTiles` 缓存池。
    *   **场景**：用户向右平移地图。
    *   左边的瓦片移出屏幕（但数据保留在缓存）。
    *   右边新出现了一排瓦片。
    *   **动作**：前端计算发现，只有右边这排新瓦片是 "未加载" 的，于是 **只针对这些新瓦片** 发起 API 请求。中间大部分区域因为已经有了，**完全不发请求**。

3.  **结果对比**

| 场景 | 现状 (Viewport Fetch) | 优化方案 (Tile Cache) |
| :--- | :--- | :--- |
| **原地微调地图** | 触发请求 (虽然数据大体相同) | **0 请求** (都在缓存瓦片内) |
| **左右平移** | 触发请求 (拉取整个新视野) | **1 次小请求** (只拉取新露出的边缘) |
| **来回查看** | 多次重复请求 | **0 请求** (之前加载过) |
| **Cloudflare 用量** | 高 (每次操作都耗额度) | **极低** (仅首次探索新区域消耗) |

### 实施建议

这是一个 **性价比极高** 的改动：
1.  **后端**：新增一个 API `GET /api/drawings/tile?x=...&y=...&z=...`，根据瓦片坐标查询数据库（D1 支持空间查询，或者简单算一下经纬度范围）。
2.  **前端**：改造 `useSync` 或 `MapCanvas`，引入 `TileCache` 逻辑。
3.  **时效性**：既然不要求秒级实时，可以设置缓存有效期（如 5 分钟）。5 分钟内重复看同一区域，不发请求；超过 5 分钟才后台静默刷新。


这个方案完美平衡了 **"操作流畅度"** 和 **"服务器成本"**。用户感觉不到刷新延迟（因为大部分是读缓存），而您的 Cloudflare 免费额度也能支撑大得多的用户量。

## 5. 架构简化：能否移除 `cf-workers` 项目？

**答案：可以。**

如果您决定放弃 WebSocket 实时同步，转而使用 "瓦片化缓存 + 定时/触发刷新"，那么 `cf-workers` 项目（原本用于承载 Durable Objects 和 WebSocket）将变得 **完全多余**。

### 可行性验证
1.  **数据写入**：目前的 `web` 项目中，`POST /api/drawings` 已经实现了直接写入 D1 数据库。目前的架构其实是 "双写"（API 写一份，WebSocket -> Queue 写一份）。移除 Workers 后，保留 API 写入即可。
2.  **数据读取**：`GET /api/drawings` 也是直接读 D1。
3.  **技术栈统一**：移除 `cf-workers` 后，您的项目将变成纯粹的 **Next.js (Pages/App Router) + Cloudflare D1** 架构。所有逻辑都在 `web` 目录下，部署和维护都更简单。

### 迁移步骤
1.  **停止/删除** `cf-workers` 目录。
2.  **前端改造**：
    *   移除 `SyncManager` 中的 WebSocket 连接代码 (`DurableObjectClient`)。
    *   保留并增强 API 轮询/瓦片请求逻辑。
3.  **配置更新**：确保 `web` 项目的 `wrangler.toml` 绑定了 D1 数据库（目前应该已经绑定了）。

这会让架构变得非常清爽带来的好处：
*   **开发成本降低**：不用同时启动两个 dev server。
*   **调试更容易**：不用担心跨端口通信问题。
*   **完全免费**：纯 Workers/Pages 托管 Next.js + D1 是完全在 Cloudflare 免费额度内的（只要控制好 D1 读写量）。
